LLM Response Cache IMPLEMENTED

All LLM calls go through a disk cache in data/cache/. Same prompt + model = same response, instantly. Spec: DS015.

How It Works

  1. Before calling the LLM, compute SHA-256(model + "\n" + prompt)
  2. Check if data/cache/{hash}.json exists
  3. If yes → return cached response (0ms, zero cost)
  4. If no → call LLM, save response to cache file

Cache File Format

{
  "mode": "test-fast",
  "prompt": "You are a normalization engine...\n\nCurrent request:\nCompare X and Y",
  "response": "## Intent Group 1\nAct: compare\nIntent: Compare X and Y\nOutput: Comparison",
  "cachedAt": "2026-03-27T12:00:00Z"
}

Each file contains the full request and response — you can inspect them to debug LLM behavior.

What Gets Cached

ScenarioCached?
Successful, non-empty LLM response✅ Yes
LLM error (timeout, 429, network)❌ No
Empty response❌ No
Validation failure (bad CNL)✅ Yes (the raw LLM output is valid text, just bad CNL)

Cache Stability

To ensure the same question produces the same cache key across server restarts:

  • Session IDs are excluded from synthesis prompts
  • Source IDs are normalized (src-xxx → src-REF) in synthesis prompts
  • Intent normalization and session context prompts don't contain session/source IDs

Performance Impact

ScenarioWithout cacheWith cache
Full eval (48 questions)~2 minutes~2 seconds
Single question (llm-assisted)~3 seconds~50 ms
KB ingest (story)~10 seconds~100 ms

Configuration

// config/llm.json
{
  "cache": true,           // set false to disable
  "cacheDir": "data/cache" // override path
}

Clearing the Cache

rm -f data/cache/*.json

Clear when: prompts change, LLM model changes, you want fresh non-deterministic results.
