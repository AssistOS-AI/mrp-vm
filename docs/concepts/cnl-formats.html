CNL Formats — Intent & Context IMPLEMENTED

Controlled Natural Language (CNL) is MRP-VM's internal representation — structured Markdown that is both human-readable and machine-parseable. All knowledge flows through CNL. Specs: DS004, DS005, DS007.

Intent CNL — What the User Wants

Produced by the Normalizer from raw NL. Each request becomes one or more Intent Groups:

## Intent Group 1
Act: compare
Intent: Compare BM25 and dense retrieval for CPU-only deployment
Context: We deploy on CPU only
Criterion: Latency, cost
Evidence: Current system uses BM25
Output: Structured comparison with recommendation
FieldRequiredDescription
ActYesPragmatic act from enum (see below)
IntentYesThe action or question in English
ContextNoConditions, constraints from the request
CriterionNoEvaluation criteria
EvidenceNoObservations from the input
OutputYesExpected result type

Context CNL — What the System Knows

Each piece of knowledge is a Context Unit with provenance:

## Context Unit src-001::chunk-002::unit-000
SourceId: src-001
ChunkId: src-001::chunk-002
Role: Comparison
Topic: BM25 and dense retrieval
Claim: BM25 has lower CPU cost in lexical retrieval.
Condition: CPU-only deployment.
UtilityActs: compare, recommend
Hash: a1b2c3d4...

Validation Rules

The validator (DS007) checks — fully symbolic, no LLM:

  • Required fields present (Act, Intent, Output for intents; SourceId, ChunkId, Role, Topic for context)
  • No unknown fields
  • Act value in the 10-act enum
  • Role value in the 9-role enum
  • Claim and Procedure are mutually exclusive
  • Heading format: ## Intent Group N or ## Context Unit ID
  • Sequential group numbering (1, 2, 3...)
