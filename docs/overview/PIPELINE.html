The Lifecycle of an Intent

From Raw Language to Grounded Answer

The MRP-VM pipeline is a deterministic sequence of transformations. See DS002 for the full spec, or Engine Detail for the 12-phase breakdown.

Phase 1: Intent Normalization and Validation

The pipeline begins with Neural Normalization (DS006). The VM rewrites the request into Intent CNL:

## Intent Group 1
Act: compare
Intent: Compare BM25 and dense retrieval for lexical search.
Criterion: Fast response time, low memory.
Output: Comparative recommendation.

The processing mode determines whether this uses an LLM or rule-based heuristics. The validator (DS007) checks structure, enums, and required fields — fully symbolic, no LLM.

Phase 2: Intent Decomposition

Once validated, the intent is symbolically decomposed (DS011) into targets, criteria, and a ContextProfile that guides retrieval: needed roles, query terms, act boost.

Phase 3: Evidence Selection

The VM executes a Retrieval Plan across session contexts and the Knowledge Base. Two strategies work together:

  • BM25 — exact lexical matching with per-field weighting
  • HDC/VSA — structural pattern matching with 4096-bit hypervectors

The active KB plugin determines which strategies run and how results are fused. The built-in defaults are kb-fast, kb-balanced, and kb-thinkingdb. See Retrieval Algorithms for formulas and details.

Phase 4: Regulated Execution (Plugins)

If an intent matches a plugin's capabilities, the plugin is spawned as a separate process. It receives the resolved evidence on stdin and returns a result on stdout in CNL format. See Plugin Development Guide.

Currently single-pass only. Agentic continuation loops are proposed but not yet implemented.

Phase 5: Synthesis and Grounding

Final response generation (DS017) under a strict grounding policy: the answer must be based only on retrieved evidence. No hallucination, no unsupported claims. The response is structured Markdown grouped by intent.

Session Commit

Only after success: user message + assistant response are logged, context units are added to the session store with hash-based deduplication.
