Session Lifecycle & Conversation State IMPLEMENTED

Sessions are the unit of conversation in MRP-VM. Each session owns a message log, a temporary context store, and preferences. Sessions are in-memory only — no database, no cookies. Spec: DS019.

Session Creation

A session is auto-created when a POST /chat/completions arrives without session_id. In the browser, each tab holds its own session in a JS variable — no localStorage, no cookies. Refresh = new session.

Turn Preparation

  1. Resolve or create session
  2. Check expiresAt < now → delete if expired
  3. Append any system message to session system prompt
  4. Extract last user message as currentMessage
  5. Build bounded historyForPrompt (max 20 messages, 16K chars)
  6. Resolve processing mode and retrieval profile

Session Context Store

Previous user turns are analyzed to extract reusable facts into a temporary in-memory KB.

Enters the storeRejected
Factual statements ("X is Y")Questions ("What is X?")
User preferencesCommands ("Please do X")
Environmental constraintsAssistant responses
Assumptions, definitionsTask lists

Example session context unit:

## Context Unit session::turn::unit-000
SourceId: session
ChunkId: session::turn
Role: Constraint
Topic: Deployment environment
Claim: We deploy on CPU only.
UtilityActs: verify

Turn Commit

Only after a successful response:

  1. Append user + assistant messages to messageLog
  2. Add context units to sessionContextUnits with hash-based deduplication — if a unit with the same hash exists, skip it
  3. Update sessionIndex (BM25 + HDC/VSA)
  4. Update lastActivityAt and expiresAt

If the turn fails, nothing is committed.

Expiration

Sessions expire after 30 minutes of inactivity (configurable). Checked lazily on access + periodic sweep every 60s. Expired sessions return SESSION_EXPIRED and cannot be revived. The UI auto-retries with a new session transparently.
