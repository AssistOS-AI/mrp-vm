Plugin Development Guide IMPLEMENTED

1. What is a Plugin

A plugin is an external process that receives a resolved intent bundle on stdin (CNL Markdown) and returns a processing result on stdout. Plugins can be written in any language (Node.js, Python, Rust, etc.) as long as they follow the wrapper convention. They complement retrieval with specialized processing — they do not replace it.

2. Directory Structure

wrappers/<plugin-name>/
├── manifest.json       # Plugin metadata and dispatch config
├── wrapper.js          # Entry point (or wrapper.py, etc.)
└── README.md           # Plugin documentation

The engine auto-discovers plugins by scanning wrappers/ at boot. Each subdirectory must contain a valid manifest.json.

3. manifest.json Reference

FieldTypeRequiredDescription
namestringYesUnique plugin identifier
versionstringYesSemver version
protocolVersionnumberYesI/O protocol version (must be 1)
capabilitiesstring[]YesCapabilities this plugin provides (e.g. "logical-constraint", "sat-check")
keywordsstring[]YesKeywords for keyword-based dispatch matching
commandstringYesExecutable to run (e.g. "node"). No .. or absolute paths allowed
argsstring[]YesArguments passed to the command (e.g. ["wrapper.js"])
timeoutnumberYesMax execution time in ms
prioritynumberYesDispatch priority (higher = more preferred)
exclusivebooleanNoReserved for future multi-plugin orchestration; ignored in v1
maxInputSizeBytesnumberYesMax stdin payload size. Exceeding returns PLUGIN_INPUT_TOO_LARGE
descriptionstringNoShort human-readable description

Example:

{
  "name": "z3-solver",
  "version": "0.1.0",
  "protocolVersion": 1,
  "capabilities": ["logical-constraint", "sat-check"],
  "keywords": ["constraint", "satisfiable", "prove"],
  "command": "node",
  "args": ["wrapper.js"],
  "timeout": 30000,
  "priority": 10,
  "exclusive": false,
  "maxInputSizeBytes": 65536,
  "description": "Z3 SMT solver wrapper"
}

4. Protocol I/O IMPLEMENTED

Input (stdin)

The plugin receives resolvedMarkdown from DS012 — a Markdown document containing the normalized intent and all evidence layers. Encoding: UTF-8. Terminated with EOF (close stdin).

## Resolved Intent Group 1
Act: verify
Intent: Verify that constraint X is satisfiable.
Output: Verification result.

### Current-Turn Context
#### sess-abc123::turn-003::unit-000
Role: Constraint
Claim: Variables are integers.

### Session Context
#### sess-abc123::turn-001::unit-000 (score: 0.74)
Role: Condition
Claim: All values must be positive.

### Persistent KB Context
#### src-001::chunk-000::unit-000 (score: 0.87)
Role: Constraint
Source: rules.md
Claim: a + b must equal c.

Output (stdout)

CNL Markdown with the processing result:

## Plugin Result
Status: success
Plugin: z3-solver
Confidence: high
Result: Constraint is satisfiable.
Evidence: Z3 returned SAT in 12ms.

Output fields:

  • Status — required: success | error
  • Plugin — required: plugin name
  • Confidence — required on success: high | medium | low
  • Result — required on success
  • Evidence — optional

Errors (stderr)

Structured JSON, one line:

{"code":"PARSE_ERROR","message":"...","details":{}}

Exit Codes

CodeMeaning
0Success
1Processing error
2Invalid input
3Internal timeout

5. Plugin Selection IMPLEMENTED

The engine evaluates dispatch criteria in priority order for each resolved intent group:

  1. Explicit field — If the Normalizer emits an Interpreter: z3 field in the Intent CNL, that plugin is selected directly.
  2. Act → capability mapping — The pragmatic act from the intent is matched against capabilities declared in manifests, using the canonical act→capability taxonomy from DS004.
  3. Keyword matching — Keywords from the intent text are matched against keywords in the manifest.

Conflict Resolution

When multiple plugins match the same intent:

  1. Select the plugin with the higher priority value.
  2. On tie: alphabetical order by name (deterministic tie-break).
  3. The chosen plugin is final — no automatic fallback chain.

6. Testing IMPLEMENTED

Each wrapper must be independently testable via stdin/stdout:

cat test-input.cnl.md | node wrapper.js

Create a test-input.cnl.md file in your wrapper directory with a sample resolvedMarkdown payload and verify the output matches the expected format.

7. Security IMPLEMENTED

  • Allowlist — Only plugins listed in config/engine.json → pluginAllowlist can be executed.
  • Timeout — Per-plugin timeout from manifest or engine config (default 30s).
  • Memory limit — Configurable via pluginMemoryLimitMB (default 256 MB).
  • Input size — Serialized resolvedMarkdown is checked against maxInputSizeBytes before spawning. Exceeding returns PLUGIN_INPUT_TOO_LARGE.
  • Path sanitization — command cannot contain .. or absolute paths.
  • Output validation — Plugin stdout must parse under the DS016 contract; invalid stdout becomes PLUGIN_INVALID_OUTPUT.
  • No OS-level sandboxing in v1 — plugins are trusted components.

8. Limitations

  • Single plugin per intent group — no chaining or multi-plugin orchestration in v1.
  • No continuation loops — a plugin runs once and returns.
  • See Agentic Loops for planned multi-step plugin orchestration.
